## V4L2 M2M architecture

The V4L2 M2M (memory-to-memory) interface is the Linux kernel's standard API for hardware video codecs on ARM SoCs. A codec device exposes two buffer queues: OUTPUT (raw frames in, or bitstream in for decode) and CAPTURE (bitstream out, or raw frames out for decode). The application negotiates formats, allocates buffers, and shuttles data between the queues via `VIDIOC_QBUF`/`VIDIOC_DQBUF` ioctls.

The Raspberry Pi's `bcm2835-codec` driver implements the stateful codec variant, where the driver manages bitstream parsing and the decoded picture buffer internally.

## Raw ioctl implementation

The encoder and decoder use raw `libc` ioctls rather than the `v4l2r` Rust crate. Multiple attempts to use `v4l2r`'s high-level API failed because the bcm2835 driver's non-standard buffer negotiation, specific control ordering, and NV12 plane layout differences could not be accommodated within the crate's abstractions. The raw ioctl approach gives direct control over `VIDIOC_REQBUFS`, `VIDIOC_QBUF`/`VIDIOC_DQBUF`, and `VIDIOC_STREAMON` sequencing.

## bcm2835 driver quirks

These quirks were discovered during on-device testing and are specific to the bcm2835-codec driver:

**Level 1.0 default**: the driver defaults to H.264 Level 1.0, which caps resolution at 128x96. The level must be set explicitly via `V4L2_CID_MPEG_VIDEO_H264_LEVEL` before `VIDIOC_S_FMT`, or the encoder fails for any resolution above SQCIF. The encoder auto-selects the level based on resolution: Level 3.0 for SD (up to 720x576), Level 3.1 for 720p, Level 4.0 for 1080p.

**`repeat_sequence_header` instead of `PREPEND_SPSPPS_TO_IDR`**: the standard V4L2 control `V4L2_CID_MPEG_VIDEO_PREPEND_SPSPPS_TO_IDR` is not supported. The driver uses `V4L2_CID_MPEG_VIDEO_REPEAT_SEQ_HEADER` to repeat SPS/PPS with each IDR frame.

**YU12 decoder output**: the decoder outputs YU12 (I420, planar Y + U + V), not NV12 (semi-planar Y + UV). The decoder checks the negotiated pixel format and handles both layouts.

**O_NONBLOCK required**: the device must be opened with `O_NONBLOCK` for proper poll/select behavior.

**Control ordering**: profile and level controls must be set before format negotiation (`VIDIOC_S_FMT`). Setting them after causes silent failures where the driver accepts the format but encodes at the wrong level.

## Device paths

The bcm2835 driver registers at `/dev/video11` (encoder) and `/dev/video10` (decoder). These paths are not standardized across SoCs. The `V4L2_ENC_DEVICE` and `V4L2_DEC_DEVICE` environment variables override the default paths for testing on other hardware.

## Performance

Measured on Raspberry Pi Zero 2 W with `gpu_mem=128`:

| Resolution | Encode fps | Decode fps |
|-----------|-----------|-----------|
| 640x360 | 138 | 60 |
| 1280x720 | 53 | 40 |
| 1920x1080 | 23 | 9 |

The encoder is consistently faster than the decoder. At 640x360 (the target resolution for the Pi Zero use case), the hardware comfortably handles 30 fps encode and decode simultaneously.

## Cross-SoC portability

The V4L2 M2M API is a kernel standard, but individual drivers differ in level defaults, SPS/PPS control names, output pixel formats, and device paths. Code that works on bcm2835 may need adjustments for Rockchip (hantro, rkvdec), Samsung Exynos (s5p-mfc), Allwinner (cedrus), or MediaTek drivers.

The stateless V4L2 decoder variant (used by Rockchip RK3588, Allwinner, and MediaTek SoCs) requires the application to manage bitstream parsing and the decoded picture buffer. This is a fundamentally different programming model from the stateful variant used by bcm2835. A standalone stateless decoder using raw ioctls (matching the encoder approach) is the likely path forward for those platforms.

## Camera capture

Camera frames on the Pi come from `rpicam-vid` piped as raw H.264 to stdin, which the demo publishes directly as pre-encoded video via `PreEncodedVideoSource`. A future improvement would integrate `libcamera` as a library for native frame capture with format negotiation, eliminating the subprocess.
