# Worklog: iroh-live side of the svc media relay integration

Started: 2026-04-23 ~22:00 CEST
Companion repo: `~/Code/rust/n0des` (overnight worklog there).

## Goal

Let an `iroh-live-relay` deployed from the n0des dashboard enforce
JWT auth with prefix-scoped publish/subscribe caps, and let `irl
publish`/`irl record` reach that relay via iroh-native transport
while still carrying per-token URL state.

## Shipped

1. `iroh-moq::MoqSession::connect_h3` and `iroh_moq::Moq::connect_h3`:
   WebTransport-over-iroh handshake that exposes the URL (path +
   `?jwt=<token>`) to the remote. H3 sessions bypass the actor's
   dedup so tokens/paths do not share session state, but stay
   registered with the actor for cross-session broadcast forwarding.
2. `iroh-live::relay::RelayTarget` + `Live::connect_relay` +
   `Live::subscribe_from_relay`: wrap the endpoint id, path and JWT
   and give callers one-line access to a session or subscription.
3. `iroh-live-relay`:
   - `--auth-key <path>` opts into JWT auth via `moq_relay::
     AuthConfig::key`. When unset we keep the permissive dev mode.
   - `/metrics` Prometheus endpoint exposes connection and broadcast
     counters so operators can scrape without a separate exporter.
4. `iroh-live-cli`:
   - `irl publish` gains `--relay <endpoint_id>`, `--relay-path
     <path>`, `--api-key <jwt>`; publishing via the relay now goes
     through the H3 path.
   - `irl record` mirrors the same flags and subscribes via H3,
     so the reverse flow (relay → iroh subscriber) works.
5. Cross-reviewed rooms+gossip+relays design doc at
   `plans/rooms-gossip-relays.md` with an adversarial pass and
   commit strategy for a future implementation session.

## Decisions

- HS256 single-key JWK per relay keeps the MVP tight. The relay
  already supports `--auth-key-dir` for future JWKS work.
- We intentionally did not change moq-lite's on-wire protocol.
  Auth flows through WebTransport URL query params, which every
  transport stack we care about already supports.
- No moq-lite version bump. `moq-lite-03` on the client and
  server negotiated down to `-02` where needed without any
  protocol changes.

## Non-decisions / known gaps

- `irl publish --no-serve` is mandatory for the H3-to-relay path
  right now because self-serving publishers still race the relay
  session on startup. The e2e uses `--no-serve` and documents the
  constraint. A future refactor should let the two coexist.
- Large file imports (`bbb.fmp4`) fail because the test file is not
  fragmented MP4. The e2e sidesteps this with `--test-source`.
- Rooms + relay integration is a separate workstream tracked in
  `plans/rooms-gossip-relays.md`.

## Summary

The relay piece of the n0des media-relay story is complete: the
server accepts JWT-authenticated H3 connections, the client
supports both publish and subscribe via H3, and the wire-level
round trip is proven by the svc-side e2e specs.
